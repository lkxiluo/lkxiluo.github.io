使用CocoaPods和svn共同管理依赖库

发表于 2017-03-18 |

我们在开发过程中经常会需要依赖第三方库或我们自己的私有库（本地库），关于依赖第三方库和Cocoapods的使用这里就不做赘述了，大家可以看唐巧的《用CocoaPods做iOS程序的依赖管理》这篇文章，我这里只记录我使用CocoaPods和svn共同管理私有库的过程，svn管理工具我使用的是Cornerstone。

接下来我将描述如何使用CocoaPods和svn共同管理依赖库

操作说明

  1. 创建一个框架命名为QHLUtil，再添加一些自己需要的类到QHLUtil中，编译顺利完成后，使用Cornerstone将QHLUtil项目导入已创建的文件目录下，结果如图:
  2. 打开终端，执行
    cd ~/Documents/ProjectManagerTest/QHLUtil
    ~/Documents/ProjectManagerTest/QHLUtil这是我创建项目的路径，根据你创建项目的路径进行修改
  3. 创建项目的podspec文件，执行
    pod spec create QHLUtil
    创建成功后终端显示Specification created at QHLUtil.podspec

  4. 编辑QHLUtil.podspec，执行

    vim QHLUtil.podspec

    编辑相关信息，我信息内容如下：

    Pod::Spec.new do |s|
s.name         = "QHLUtil"
s.version      = "0.0.1"
s.summary      = "工具类"
s.description  = <<-DESC
                "工具类"
                DESC

s.homepage     = "https://lkxiluo.github.io/"
s.license      = "MIT"
s.author       = { "Qianhan" => "https://lkxiluo.github.io/" }
s.platform     = :ios, "8.0"
s.ios.deployment_target = "8.0"
s.source       = { :svn => "svn://192.168.6.5:9600/wzhouhui/app/ios/ProjectManagerTest/QHLUtil", :tag => "#{s.version}" }
s.source_files  = "QHLUtil", "QHLUtil/**/*.{h,m}"
s.exclude_files = "QHLUtil/Exclude"
s.frameworks = "UIKit", "Foundation"

    关于.podspec文件内容各个字段的代表的意思这里就不描述了，大家可以看这篇文章
    保存退出，将QHLUtil.podspec文件同步到svn上

  5. 创建测试项目ProjectManagerDemo，执行
    cd /Users/karl.luo/Documents/ProjectManagerTest/ProjectManagerDemo
    到你创建的项目文件目录下，这里的命令根据你的项目路径做改变

  6. 创建Podfile文件，执行

    vim Podfile

    编辑Podfile文件内容如下

    platform:ios, '8.0'

target:ProjectManagerDemo do

pod 'QHLUtil', :svn=>'svn://192.168.6.5:9600/wzhouhui/app/ios/ProjectManagerTest/QHLUtil'

end

    svn://192.168.6.5:9600/wzhouhui/app/ios/ProjectManagerTest/QHLUtil为你私有库文件目录

  7. 建立依赖关系，这里有两种情况

    a. 直接执行命令

    pod install

    完成依赖关联
    b. 根据版本进行管理
    进入QHLUtil文件目录，创建名为tags的文件，如图：

    将tags文件夹同步到svn，打开Cornerstone，右击QHLUtil项目，创建改项目的一个版本tag，如图：

    填写tag的信息

    Tag信息说明：
    Tag As为版本号，将与QHLUtil.podspec文件中的s.version= “0.0.1”相同
    Where为保存路径，保存在刚才创建的tags文件夹下，此目录要正确，因为关联时获取资源就是根据s.source= { :svn => “svn://192.168.6.5:9600/wzhouhui/app/ios/ProjectManagerTest/QHLUtil”, :tag => “#{s.version}” }索引的，如果错了见找不到资源
    Revision为svn上代码的提交版本
    点击“Create Tag”创建完成，此时，Cornerstone中项目的目录结构将如图所示:

    编辑QHLUtil.podspec文件（步骤同上）,将s.version值设置为刚才创建的Tag As号，当有多个tag时，设置为你要选择的版本号，保存退出编辑
    执行

    pod spec lint QHLUtil.podspec --allow-warnings

    检查信息是否有错，如果成功，来到ProjectManagerDemo目录，执行

    pod install

  8. 更新新库，需要更新依赖库时，编辑QHLUtil.podspec文件中s.version设置为要依赖的库，执行

    pod update

我的iOS编码规范

发表于 2017-03-10 |

规范的级别

  • 强制使用(A)
  • 强烈推荐(B)
  • 可选性(C)
  • 不建议(D)

命名规范

任何命名都要准确表达其用途(B)

类名的命名规范

1.大驼峰命名法(A)
类名的所有单词首字母必须大写
2.类名加前缀(B)
为防止与其他依赖类库的冲突，苹果享有两位前缀的使用权，所以前缀一般为三个大写字母，比如说作者名首字母大写，公司名首字母大写，Category雷同
3.类名一般以其继承的类作结尾(B)

例如
QHLUserClass, NSString+QHLUserClass, QHLEditButton, QHLUserTableViewCell, QHLUserDetailViewController

方法的命名规范

1.小驼峰命名法(A)
方法名和参数名的首字母小写，其余单词的首字母大写
2.方法名要清晰、简明，使用的单词能正确表达方法的作用(B)
3.Category的方法名前要加前缀，防止引用冲突(B)
一般以两个或三个小写字母加“_”为前缀，本人使用三个小写字母，与类名的命名一致
4.方法名前“-”、“+”与返回值间有一个空格，方法实现开始的”{“在方法名的下一行(C)
5.当参数过长时，参数换行，以“:”对齐(C)
6.参数变量的声明类型与”*”以一个空格隔开(C)

例如
\- (void)deleteUserWithId:(NSInteger)userId;
\- (BOOL)qhl_isPhoneNumber
  {
     return YES;
  }
\- (void)initUserWithId:(NSInteger)userId
                   name:(NSString *)userName
                    age:(NSInteger)age
                 gender:(NSString *)gender;

(property)属性的命名规范

1.小驼峰命名法(A)
属性名首字母小写，其余单词的首字母大写
2.属性类型如有“*”，则“*”靠近属性名，变量名左对齐(C)
3.属性名以声明的类型能表达其类型的单词结尾(B)
4.变量名遵循属性的命名规范, 同时以”_”为前缀(B)

例如
@property (nonatomic, strong) UIView    *childContaintView;
@property (nonatomic, strong) UIButton  *editButton;
@interface QHLUserClass () {
    UITextField  *_userNameTextField;
}

(protocol)协议的命名规范

1.大驼峰命名法，与类命名规则一样(A)
2.协议名加前缀，与类命名规则一样(B)
3.一般的协议名以Protocol结尾(B)
4.委托代理

a.如果协议的作用是委托类获取数据则协议名形式为“类名+DataSource”(B)
b.如果协议的作用是委托类实现操作则协议名形式为“类名+Delegate”(B)
c.协议中的方法名参数必须有一个是委托类(B)

例如 
@protocol QHLUserModelProtocol <NSObject>
@end
@protocol UITableViewDataSource <NSObject, UIScrollViewDelegate>
\- (NSInteger)numberOfSectionsInTableView:(UITableView *)tableView;
@end

枚举的命名规范

1.所属类名+作用(B)
2.定义在.h文件时，枚举成员以枚举名+成员名(B)
3.定义在.m文件时，枚举成员以小写”k”为前缀+成员名(C)
4.声明枚举时使用NS_ENUM、NS_OPTIONS而不使用enum(B)
5.当代码实现中使用到(if或switch)0、1、2、3···等数字进行判断是，使用枚举代替（C）

例如
typedef NS_ENUM(NSInteger, QHLUserClassGender) {
    QHLUserClassGenderUnknow = 0,    // 未知性别
    QHLUserClassGenderMan,           // 男性
    QHLUserClassGenderwomen          // 女性
};

宏或全局常量命名规范

1.使用固定值时，优先使用常量定义，再使用宏(C)
2.声明在.h文件的常量以该(文件)类类名为前缀+每个单词首字母大写的常量名(B)
3.声明在.m文件的常量以小写”k”为前缀+每个单词首字母大写的常量名(C)
4.宏定义的所有字母都为大写，也以该(文件)类类名为前缀，并且每个单词以”_”隔开(B)
5.一些特殊的宏或常量以特殊单词结尾，比如说通知以NOTIFICATE或Notificate结尾

例如
NSString * const QHLUserClassGender = @"男";
\#define QHLUSERCLASS_GENDER     @"男"
\#define QHLUSERCLASS_UPDATE_USERNAME_NOTIFICATE @"QHLUSERCLASS_UPDATE_USERNAME_NOTIFICATE"

图标命名规范

1.图标都以小写字母命名，各个单词间用”“隔开(B)
2.图标的命名结构为:“功能模块名作用名_状态”，正常状态用”normal”，高亮用”hl”，两倍图加”@2x”，三倍图加“@3x”(B)
例如
home_edit_normal@2x.png为首页正常状态的2倍编辑图标

注释

1.类和方法的注释以Xcode自带的注释格式为准(C)
2.单行的注释以“//”+注释内容，位置为该行的上方(C)
3.多行(代码块)使用“/ 注释内容 /”注释，位置为代码块第一行的上方(C)
4.属性、变量、枚举怎在相应行的结尾以“//”+注释内容注释(C)
5.对未实现、待实现的内容以”//TODO:”+注释内容标注(C)
6.功能块以”#pragma mark - “+注释，比如“#pragma mark - getter/setter”(C)
7.子功能块以”#pragma mark -“+注释(C)

例如

#pragma mark - delegate

#pragma mark -TableViewDelegate

8.宏和全局常量则在其上面以”/ 注释内容 /“进行注释

代码风格

1.每个代码块间各一行(C)
2.不管代码的多少，if、switch语句中都使用”{}”，并且”{“隔一个空格接在第一行之后，每个分支以一行间隔(C)

例如
- (NSString *)getUserName
{
    if (条件) {
        return @"";

    } else {
        retrun @"";
    }
}
